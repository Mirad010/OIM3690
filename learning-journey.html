My Semester Building the Web

Ten weeks, logged commit by commit — from a bare hello.html to a hotel investment calculator running on real economic data. This page is built from my own weekly logs.

Concepts, in the order I actually learned them

wk1HTML tags wk1git add / commit / push wk3CSS selectors wk3the cascade wk4Flexbox wk4CSS Grid wk4HTML forms, GET vs POST wk5JS events, onclick wk6DOM manipulation wk6localStorage wk7looping over arrays of data wk7async fetch, POST requests wk8consuming a live API wk8handling a dead API wk9chart + map libraries wk10combining multiple data sources

wk01 · foundations

Set up both repos, wrote my first HTML by hand

Built

Set up oim3690 and mirad010.github.io on GitHub, then wrote hello.html by hand.

Learned

Basic HTML tags, and how to actually push code with git add / commit / push.

AI

Asked AI to add a nav bar, a section, and a footer to hello.html.

Didn't understand yet

How CSS styling works, or what Live Server actually does.

wk02 · rewriting to understand

Took AI's code apart and rebuilt the nav bar myself

Built

Rewrote the nav bar styling from wk01 and swapped in my real interests instead of placeholder content.

Learned

Why I changed it: I wanted to actually understand how nav, section, and footer tags work instead of just having AI generate them.

wk03 · first real CSS

Wrote style.css from scratch — selectors, cascade, DevTools

Built

css/style.css with body, h1, .highlight, and a:hover rules, linked into about-me.html.

Learned

How element, class, and pseudo-class selectors work, how styles cascade, and how to use DevTools to inspect what's actually being applied.

AI

Used AI to understand CSS syntax and selector logic.

Didn't understand yet

How specificity actually gets calculated, or what Flexbox does.

wk04 · MP1

Built YOYO Clothing — a real site for a friend, first forms

Built

MP1: a clothing site for my friend Ali, with product photos, a contact.html form, and CSS to style it.

Learned

Flexbox and Grid for layout, how HTML forms work (action, method, input types), and the difference between GET and POST.

AI

Used AI to structure the site, build the contact form, and write PRD.md for the project.

Didn't understand yet

Responsive design on mobile, and how JavaScript would eventually connect to forms.

wk05 · JavaScript shows up

Added my first real interactivity — a modal that opens on click

Built

js-demo.html, and a JavaScript modal popup on the YOYO shop page.

Learned

That JavaScript can make a page interactive without reloading it, and how onclick triggers a function.

AI

Used AI to add the modal popup and redesign the home and contact pages.

Didn't understand yet

What document.getElementById() actually does, why addEventListener exists instead of just onclick, and where console.log() output even shows up.

wk06 · MP2

Wrote real functions by hand, finished Habit Tracker Pro

Built

A to-do app with localStorage, a two-way temperature converter, a discount calculator with a real getDiscount() function, and finished MP2.

Learned

Writing functions with if/else by hand, converting input with Number()/parseFloat(), updating the DOM without alert() boxes, and how localStorage persists data between sessions.

AI

Used AI to scaffold the to-do app and then reviewed the code myself; used it to turn a one-way converter into two-way, and a basic function into a full checkout page.

wk07 · data and APIs begin

Rendered data from an array, sent my first fetch request

Built

playlist.html (array of song objects rendered with a for loop), post-message.html (async POST request with an auth header), and fetch-fun.html (Dog CEO + Advice Slip APIs).

Learned

The render-from-data pattern — data lives in one place, a loop builds the page. Also that different APIs bury their data in different JSON fields (data.message vs data.slip.advice), so you actually have to go look.

wk08 · MP3

Finished Country Explorer — then watched the API die mid-project

Built

MP3, Country Explorer: search, region filter, sorting, live capital-city weather, a compare mode, a flag quiz, and favorites in localStorage.

Learned

That a real API can just stop working while you're using it. REST Countries v3.1 got deprecated mid-build and v5 needed a paid key I couldn't put in a public repo — switched to countries.dev instead and remapped every field name. The error state I'd already built (instead of letting it crash) is what kept the site usable while I fixed it.

AI

AI wrote the original fetch code against REST Countries; when it broke in production, I diagnosed the real cause by opening the API URL directly in the browser, then rewrote it against countries.dev myself.

wk09 · final project starts

Started Hotel Feasibility Studio — my family's actual property question

Built

The first working version of my final project: sliders for rooms, cost, rate, and occupancy that recalculate live, a break-even bar, a 15-year cash flow chart, a map, and currency conversion.

Learned

That break-even analysis from ACC 2002 and payback period from finance translate into real JavaScript pretty directly once you write them out as formulas.

AI

Talked through the idea and the formulas with AI before building, then had it help scaffold the HTML/CSS/JS faster than typing all of it by hand.

Question

I checked the calculation logic and it seems right, but I want to make sure there aren't small errors in how variable costs combine with occupancy before building more on top of it.

wk10 · final project, fixed

Found the map was fake, made it actually drive the numbers

Built

Connected the map to the calculations for real: World Bank GDP-per-capita data scales construction cost and room rate to the local economy, and OpenStreetMap's Overpass API counts real hotels/attractions/restaurants near the pin to estimate occupancy. Also fixed my documentation — PROPOSAL.md, README.md, WORKLOG.md.

Learned

Free public APIs fail in specific, fixable ways — my first Overpass request returned nothing because it was a POST instead of a GET, and the main server sometimes doesn't answer at all, so I added backup mirrors.

AI

Worked through a couple of failed approaches with AI before landing on the GDP + OpenStreetMap combination, then used it to debug the Overpass request once it started failing.

Question

Is a GDP-per-capita based formula a reasonable simplification for a first version, or too big an assumption to hang real financial numbers on?

Looking back at wk01, I didn't know what a tag was for. By wk10 I was debugging a failed API request against a live financial model I built myself. The biggest shift wasn't syntax — it was going from asking AI to build things for me, to asking it to help me build things I already understood.

Final project

Hotel Feasibility Studio — a hotel investment calculator driven by real GDP and location data.
